# h_claw 人工接管滑块测试用例

## 1. 测试目标

验证系统能够在检测到交互式安全挑战时停止自动重试，保留原浏览器页面，把控制权安全地交给已授权 IM 用户，并在人工验证完成后准确、幂等地恢复原抓取任务。

测试不验证自动破解滑块能力；测试环境应使用自建挑战模拟页或平台明确允许的测试账号与测试页面。

## 2. 测试分层

- 单元测试：状态机、检测器、令牌、限流、熔断、资源所有权。
- 组件测试：ChallengeManager、数据库、接管网关、IM adapter。
- 集成测试：Playwright/浏览器 worker、WebSocket、任务暂停恢复。
- 端到端测试：IM 卡片到人工操作再到结果消息。
- 安全测试：授权、令牌、导航限制、日志脱敏、资源隔离。
- 稳定性测试：并发、超时、断线、进程关闭和重复消息。

## 3. 测试环境

### 3.1 挑战模拟站点

提供以下可控路由：

- `/product/normal`：正常商品页。
- `/product/challenge`：首次返回滑块模拟页，通过人工操作后跳回商品页。
- `/product/challenge-always`：始终返回挑战页。
- `/login`：登录页。
- `/security-check`：离开登录 URL 但仍未登录的安全验证页。
- `/product/slow`：用于超时和取消测试。
- `/redirect-external`：跳转到非允许域名。

模拟滑块只需要验证连续人工输入事件或显式测试控件，不应包含真实站点算法。

### 3.2 测试身份

- `user_a`：任务创建者。
- `user_b`：未授权用户。
- `operator`：被明确授权的运维人员。

### 3.3 默认配置

- 挑战 TTL：300 秒；自动化测试可覆盖为 2 秒。
- `max_per_domain=1`。
- 每用户最大活动挑战数：1。
- 全局最大活动挑战数：2。
- 挑战冷却时间：测试环境设为 10 秒。

## 4. 单元测试用例

### 状态机

#### UT-001 创建挑战

- 优先级：P0
- 前置：存在 `RUNNING` 抓取任务和存活页面。
- 操作：调用 `ChallengeManager.create`。
- 预期：生成唯一 `challenge_id`；状态为 `WAITING_HUMAN`；任务状态为 `HUMAN_REQUIRED`；到期时间正确。

#### UT-002 合法状态迁移

- 优先级：P0
- 操作：依次执行 `WAITING_HUMAN -> IN_CONTROL -> VERIFYING -> SOLVED`。
- 预期：每步成功；记录时间戳；最终唤醒一次恢复流程。

#### UT-003 非法状态迁移

- 优先级：P0
- 操作：尝试 `SOLVED -> IN_CONTROL`、`EXPIRED -> SOLVED`、`CANCELLED -> VERIFYING`。
- 预期：全部拒绝，状态不改变。

#### UT-004 终态幂等

- 优先级：P0
- 操作：对已 `SOLVED` 挑战重复提交 solved 回调。
- 预期：返回幂等成功或已完成；不重复唤醒、不重复抓取。

#### UT-005 自动过期

- 优先级：P0
- 操作：推进测试时钟超过 `expires_at`。
- 预期：状态变为 `EXPIRED`；页面和 context 被关闭；等待者收到超时结果。

### 挑战检测

#### UT-006 关键词检测

- 优先级：P0
- 输入：包含配置挑战关键词的 HTML。
- 预期：返回 `captcha_detected` 和脱敏的 matched signal。

#### UT-007 选择器检测

- 优先级：P0
- 输入：包含挑战 iframe/容器选择器的 HTML。
- 预期：识别为交互挑战。

#### UT-008 挑战域名重定向

- 优先级：P0
- 输入：最终 URL 为配置的挑战供应商域名。
- 预期：返回 `captcha_redirect`。

#### UT-009 正常商品页不误判

- 优先级：P0
- 输入：包含商品标题、价格和 JSON-LD 的页面。
- 预期：检测通过，不创建 challenge。

#### UT-010 登录页与挑战页区分

- 优先级：P0
- 输入：登录页、滑块页各一份。
- 预期：分别返回 `login_redirect/login_required` 与 `captcha_detected`。

#### UT-011 站点适配器隔离

- 优先级：P1
- 操作：淘宝适配器特征出现在非淘宝测试域名。
- 预期：不应用淘宝域名专用规则；通用规则仍正常工作。

### 浏览器资源所有权

#### UT-012 普通成功释放页面

- 优先级：P0
- 操作：fetcher 返回正常页面。
- 预期：转换完成后 Page/Context 各关闭一次。

#### UT-013 挑战时移交页面

- 优先级：P0
- 操作：fetcher 检测到挑战。
- 预期：fetcher 不关闭页面；ChallengeManager 获得所有权。

#### UT-014 挑战完成后释放页面

- 优先级：P0
- 操作：人工验证成功并完成恢复抓取。
- 预期：结果读取完成后页面只关闭一次。

#### UT-015 创建挑战失败时释放资源

- 优先级：P0
- 操作：模拟数据库写入或 ChallengeManager 创建失败。
- 预期：Page/Context 被关闭；任务进入明确失败状态。

### 限流、合并与熔断

#### UT-016 单域名并发限制

- 优先级：P0
- 操作：并发提交 5 个相同域名、不同 URL 请求，`max_per_domain=1`。
- 预期：任意时刻最多一个上游访问。

#### UT-017 不同域名并发

- 优先级：P0
- 操作：同时提交两个不同域名请求。
- 预期：在全局配额允许时可以并发执行。

#### UT-018 Singleflight 请求合并

- 优先级：P0
- 操作：并发提交 10 个相同缓存键请求。
- 预期：只调用一次上游 fetcher；所有调用者收到同一结果。

#### UT-019 挑战后停止自动升级

- 优先级：P0
- 操作：HTTP/browser 任一层返回明确挑战。
- 预期：不再调用后续层和 stealth retry；直接创建人工挑战或返回已有挑战。

#### UT-020 挑战负缓存

- 优先级：P0
- 操作：挑战活动期间再次请求同一 URL。
- 预期：复用已有 `challenge_id`，不产生新上游请求。

#### UT-021 冷却期阻断

- 优先级：P0
- 操作：挑战失败后在 `next_allowed_at` 之前重试。
- 预期：返回 `COOLDOWN` 和剩余时间，不访问上游。

#### UT-022 冷却期结束

- 优先级：P1
- 操作：推进时钟超过 `next_allowed_at` 后重试。
- 预期：允许创建新抓取任务。

### 接管令牌

#### UT-023 生成一次性令牌

- 优先级：P0
- 预期：令牌随机性充足；存储层只有摘要；载荷绑定挑战、用户和有效期。

#### UT-024 令牌只能使用一次

- 优先级：P0
- 操作：连续使用同一令牌两次。
- 预期：第一次成功，第二次拒绝。

#### UT-025 过期令牌

- 优先级：P0
- 操作：使用超过到期时间的令牌。
- 预期：拒绝，不暴露挑战是否存在。

#### UT-026 用户身份不匹配

- 优先级：P0
- 操作：`user_b` 使用绑定 `user_a` 的令牌。
- 预期：拒绝并记录安全审计事件。

#### UT-027 挑战 ID 篡改

- 优先级：P0
- 操作：将合法令牌用于另一个挑战 URL。
- 预期：拒绝。

### 登录成功验证

#### UT-028 跳离登录页但进入安全页

- 优先级：P0
- 操作：登录页跳转到 `/security-check`。
- 预期：不得标记登录成功，不创建 ACTIVE profile。

#### UT-029 正常登录成功

- 优先级：P0
- 操作：跳转到允许业务域名，挑战标记消失，登录态 DOM 存在。
- 预期：创建 ACTIVE profile。

#### UT-030 跳转到非允许域名

- 优先级：P0
- 操作：登录后跳转到外部域名。
- 预期：失败并关闭 context。

## 5. 组件与集成测试

#### IT-001 数据库迁移可重复执行

- 优先级：P0
- 操作：在空库执行迁移两次。
- 预期：第二次无错误，表和索引结构正确。

#### IT-002 已有数据库升级

- 优先级：P0
- 操作：从升级前 schema 执行迁移。
- 预期：既有抓取和 profile 数据不丢失。

#### IT-003 接管 WebSocket 建连

- 优先级：P0
- 操作：授权用户通过有效入口建立连接。
- 预期：看到当前原始挑战页；挑战状态为 `IN_CONTROL`。

#### IT-004 未授权 WebSocket

- 优先级：P0
- 操作：缺少登录态、Origin 不合法或用户不匹配时连接。
- 预期：握手被拒绝，不发送任何浏览器画面。

#### IT-005 连续拖动输入传递

- 优先级：P0
- 操作：在接管页面对模拟滑块执行按下、移动、释放。
- 预期：原浏览器页面收到连续输入并切换到正常商品页。

#### IT-006 缩放与坐标转换

- 优先级：P1
- 操作：在不同 WebView 尺寸和 DPR 下拖动模拟控件。
- 预期：输入映射到正确浏览器坐标。

#### IT-007 人工验证成功后原页恢复

- 优先级：P0
- 操作：完成模拟挑战。
- 预期：不重新导航目标 URL；直接从原 Page 取得正常 DOM并完成转换。

#### IT-008 人工验证未通过

- 优先级：P0
- 操作：用户操作后页面仍包含挑战标记。
- 预期：状态保持活动或返回 `VERIFY_FAILED`，不得恢复抓取。

#### IT-009 接管中用户断线重连

- 优先级：P1
- 操作：断开 WebSocket 后在允许窗口内用同一已认领会话重连。
- 预期：仍看到原页面；总 TTL 不延长；不存在两个控制者。

#### IT-010 断线直至超时

- 优先级：P0
- 操作：接管后断线且不重连。
- 预期：挑战过期，页面释放，任务失败消息送达 IM。

#### IT-011 IM 重复事件

- 优先级：P0
- 操作：重复投递相同按钮点击、webhook 和成功事件。
- 预期：只认领一次、只恢复一次、只写入一个最终结果。

#### IT-012 IM 消息发送失败

- 优先级：P0
- 操作：创建挑战后模拟 IM API 失败。
- 预期：按策略重试消息；达到上限后取消挑战并释放页面，或进入可观测待处理状态。

#### IT-013 最大活动挑战容量

- 优先级：P0
- 操作：达到全局容量后再创建挑战。
- 预期：返回 `CHALLENGE_CAPACITY_EXCEEDED`；新页面被释放；已有挑战不受影响。

#### IT-014 每用户容量

- 优先级：P0
- 操作：同一用户创建超过上限的活动挑战。
- 预期：拒绝新增；可返回已有挑战入口。

#### IT-015 Profile 串行使用

- 优先级：P0
- 操作：两个任务同时使用同一 profile。
- 预期：串行执行，不发生 profile 目录并发打开。

## 6. 端到端测试

#### E2E-001 完整成功路径

- 优先级：P0
- 步骤：
  1. `user_a` 在 IM 提交模拟商品 URL。
  2. 爬虫检测挑战并返回卡片。
  3. `user_a` 点击“打开验证”。
  4. 用户完成模拟滑块。
  5. 系统确认挑战消失并恢复原任务。
- 预期：IM 收到商品抓取成功消息；只访问一次初始目标；challenge 进入 `SOLVED`；浏览器资源最终释放。

#### E2E-002 用户取消

- 优先级：P0
- 步骤：用户点击“取消任务”。
- 预期：挑战和抓取任务均进入 `CANCELLED`；链接立即失效；资源释放。

#### E2E-003 用户未处理导致超时

- 优先级：P0
- 步骤：收到卡片后不打开。
- 预期：到期后 IM 收到超时消息；没有自动重试；资源释放。

#### E2E-004 非创建者尝试接管

- 优先级：P0
- 步骤：`user_b` 获得并打开 `user_a` 的链接。
- 预期：无法看到浏览器画面；`user_a` 的挑战仍有效；产生审计记录。

#### E2E-005 验证后仍为登录墙

- 优先级：P0
- 步骤：模拟滑块通过但页面进入登录页。
- 预期：不得输出商品成功；返回 `LOGIN_REQUIRED` 并按产品策略发起登录流程。

#### E2E-006 验证后跳转外部域名

- 优先级：P0
- 步骤：模拟挑战完成后跳转到非允许域名。
- 预期：立即停止、关闭页面并返回安全错误。

#### E2E-007 服务优雅关闭

- 优先级：P1
- 步骤：存在活动挑战时发起服务关闭。
- 预期：停止接收新挑战；活动挑战按策略完成或收到中断通知；无残留浏览器进程。

## 7. 安全测试

#### SEC-001 URL 令牌不进入日志

- 优先级：P0
- 操作：完成一次接管，并检查应用、反向代理和审计日志。
- 预期：令牌、Cookie、Authorization、二维码均已脱敏。

#### SEC-002 WebSocket Origin 校验

- 优先级：P0
- 操作：从非允许 Origin 发起连接。
- 预期：拒绝连接。

#### SEC-003 非目标域名导航

- 优先级：P0
- 操作：接管中尝试导航到外部域名、内网 IP 和云元数据地址。
- 预期：全部阻断并记录安全事件。

#### SEC-004 危险功能限制

- 优先级：P0
- 操作：尝试下载、文件上传、剪贴板、打印和打开开发者工具。
- 预期：均不可用，除非某项经过明确安全评审和授权。

#### SEC-005 并发控制劫持

- 优先级：P0
- 操作：两个客户端同时使用合法身份认领同一挑战。
- 预期：只有一个客户端获得控制权；另一个只看到“正在处理中”。

#### SEC-006 令牌重放

- 优先级：P0
- 操作：挑战完成后再次访问旧链接。
- 预期：拒绝且不泄露页面或结果详情。

#### SEC-007 会话固定攻击

- 优先级：P1
- 操作：在授权前后替换 IM/Web 登录会话标识。
- 预期：接管授权失效，必须重新完成身份校验。

#### SEC-008 页面内容注入

- 优先级：P0
- 操作：模拟目标页包含脚本，尝试访问接管网关父页面或令牌。
- 预期：浏览器隔离、CSP 和不同 origin 阻止访问。

## 8. 稳定性与性能测试

#### PERF-001 100 次生命周期无泄漏

- 优先级：P0
- 操作：循环执行成功、取消、失败、超时各 25 次。
- 预期：活动 Page、Context、WebSocket、临时目录和令牌数量回到基线。

#### PERF-002 并发挑战容量

- 优先级：P1
- 操作：以配置上限创建并维持挑战。
- 预期：CPU、内存和文件描述符在预算内；超额请求被快速拒绝。

#### PERF-003 慢客户端

- 优先级：P1
- 操作：接管客户端读取画面极慢。
- 预期：不会无限堆积帧；采用丢帧或背压；不影响其他挑战。

#### PERF-004 数据库短暂不可用

- 优先级：P1
- 操作：挑战状态更新期间暂停数据库。
- 预期：状态不会错误地标成成功；恢复后可重试或安全失败；页面最终释放。

#### PERF-005 IM 服务短暂不可用

- 优先级：P1
- 操作：验证成功时 IM API 不可用。
- 预期：抓取结果仍持久化；通知进入重试队列；不重复恢复任务。

## 9. 回归测试

#### REG-001 普通 HTTP 抓取

- 优先级：P0
- 预期：不需要浏览器的页面仍走原 HTTP 路径并正常缓存。

#### REG-002 普通动态页面

- 优先级：P0
- 预期：无挑战的 SPA 仍可通过浏览器层成功转换。

#### REG-003 登录态抓取

- 优先级：P0
- 预期：有效 profile 的正常页面无需人工接管即可抓取。

#### REG-004 SSRF 防护

- 优先级：P0
- 预期：初始 URL、最终 URL 和接管期间导航都继续执行 SSRF 防护。

#### REG-005 缓存隔离

- 优先级：P0
- 预期：公开内容缓存不泄露账号私有内容；不同 session 的私有缓存相互隔离。

#### REG-006 大结果分段读取

- 优先级：P1
- 预期：人工恢复后的大 Markdown 仍可通过原 `read_crawl_result` 分段读取。

## 11. 自动化建议

- 单元测试使用 fake clock，避免真实等待 TTL 和冷却期。
- 浏览器集成测试使用本地模拟站点，不访问淘宝或什么值得买生产环境。
- 用可计数的 fake Page/Context 断言 `close()` 恰好调用一次。
- 对恢复、消息和 webhook 使用 idempotency key，并用并发测试验证。
- CI 中区分：`unit`、`integration`、`browser`、`security`、`e2e-manual`。
- 真站人工验收只在取得授权的测试账号和受控低频条件下执行，不作为普通 CI 的一部分。

## 12. 通过标准

- P0 单元、组件、集成、端到端、安全和回归用例全部通过。
- P1 用例无阻断上线的问题，未通过项有明确风险接受记录。
- 100 次混合生命周期测试没有资源泄漏。
- 同一任务在任意重复消息和并发回调下最多恢复一次。
- 检测到明确挑战后，上游请求计数不再增加。
- 任意未授权身份都不能看到浏览器画面或发送控制事件。
