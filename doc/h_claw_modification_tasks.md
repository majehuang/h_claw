# h_claw 人工接管滑块改造任务清单

## 1. 文档目的

本文档把“滑块触发后由 IM 用户人工接管同一浏览器会话，验证完成后恢复原抓取任务”的方案拆解为可开发、可测试、可验收的任务。

改造不包含自动破解、模拟拖动、第三方打码、购买 Cookie 或通过频繁更换网络身份规避验证。系统只负责识别挑战、暂停任务、安全地把平台原有验证界面交给已授权用户，并在用户完成验证后继续任务。

## 2. 当前问题摘要

- `max_per_domain` 已存在于配置，但调度器没有执行单域名并发限制。
- `HTTP -> browser -> stealth` 自动升级可能对同一 URL 连续发起多次请求。
- 挑战失败记录立即过期，相同请求可能立刻重新触发完整抓取链。
- 浏览器 fetcher 只返回最终 HTML，没有向上层移交存活的 `Page`/`BrowserContext`。
- 登录和后续抓取可能重新启动不同的浏览器会话，环境一致性不足。
- 登录成功主要以“离开登录 URL”为依据，安全验证跳转可能被误判为成功。
- 当前挑战检测特征偏通用，缺少站点适配器和证据记录。
- MCP 请求模型缺少“暂停、异步等待、恢复”的任务状态。

## 3. 目标流程

```text
抓取任务
  -> 检测到交互式挑战
  -> 停止升级和自动重试
  -> 保留原 Page/BrowserContext
  -> 创建 WAITING_HUMAN 挑战会话
  -> IM 发送一次性接管入口
  -> 用户在 WebView/网页中操作原页面
  -> 后端确认挑战消失且目标内容恢复
  -> 撤销控制权限
  -> 从原页面恢复抓取和转换
  -> IM 推送最终结果
```

## 4. 状态模型

### 4.1 抓取任务状态

- `RUNNING`
- `HUMAN_REQUIRED`
- `RESUMING`
- `SUCCESS`
- `FAILED`
- `CANCELLED`

### 4.2 挑战会话状态

- `WAITING_HUMAN`：等待用户打开接管页面。
- `IN_CONTROL`：授权用户正在控制页面。
- `VERIFYING`：用户操作后，服务端正在确认结果。
- `SOLVED`：挑战已通过，可以恢复抓取。
- `FAILED`：挑战页面异常或验证结果无效。
- `EXPIRED`：超过允许接管时间。
- `CANCELLED`：用户或系统主动取消。

状态迁移必须使用条件更新或锁保护，任何终态都不得重新回到活动状态。

## 5. 任务清单

### P0：必须完成

#### HC-001 定义人工挑战领域模型

- [ ] 新增 `ChallengeSession`、`ChallengeStatus` 和错误码定义。
- [ ] 字段至少包含：`challenge_id`、`job_id`、`owner_id`、`domain`、`status`、`created_at`、`expires_at`、`solved_at`、`failure_reason`。
- [ ] 数据库只保存元数据和状态，不保存 Cookie、令牌、二维码、页面截图或浏览器对象。
- [ ] 明确单进程版中存活 `Page/BrowserContext` 仅保存在内存；多实例部署必须使用粘性路由或独立 Browser Worker。

验收标准：状态、字段、终态幂等性和过期行为均有单元测试。

#### HC-002 增加数据库迁移

- [ ] 新增 `challenge_sessions` 表。
- [ ] 为 `job_id`、`owner_id + status`、`expires_at` 建索引。
- [ ] 为 `crawl_results` 或独立任务表增加 `HUMAN_REQUIRED`、`RESUMING` 所需字段。
- [ ] 保存 `next_allowed_at`，支持挑战后的冷却期。
- [ ] 迁移必须可重复执行。

依赖：HC-001。

验收标准：空库、已有库和重复执行迁移均成功。

#### HC-003 重构浏览器抓取返回类型

- [ ] 将“普通抓取结果”和“需要人工接管的存活页面”表示为不同结果类型。
- [ ] 检测挑战前不得关闭 `Page/BrowserContext`。
- [ ] 普通成功、普通失败仍自动释放页面。
- [ ] 进入人工接管后，页面所有权转移给 `ChallengeManager`。
- [ ] 明确定义所有异常、取消和超时路径的资源释放责任。

建议接口：

```python
BrowserFetchResult = CompletedFetch | InteractiveChallenge
```

依赖：HC-001。

验收标准：每条路径的页面关闭次数恰好为一次，不泄漏 context。

#### HC-004 实现 ChallengeManager

- [ ] 实现 `create`、`claim_control`、`verify`、`wait`、`cancel`、`expire`、`release`。
- [ ] 同一挑战只允许一个控制者。
- [ ] 使用 `asyncio.Event`/Future 通知恢复流程，避免轮询阻塞 MCP 调用。
- [ ] 定时清理过期挑战和关联浏览器资源。
- [ ] 进程退出时关闭所有活动挑战。
- [ ] 处理用户断线后短时间重连，但不得延长总过期时间。

依赖：HC-001、HC-003。

验收标准：并发认领、取消、过期和重复回调结果确定且无资源泄漏。

#### HC-005 挑战检测与证据标准化

- [ ] 保留通用关键词、选择器和挑战域名检测。
- [ ] 增加站点适配器接口，不把淘宝/什么值得买规则硬编码进通用 detector。
- [ ] 检测结果包含 `provider`、`reason`、`final_url`、`matched_signal`。
- [ ] 证据日志不得包含完整 HTML、Cookie 或用户个人信息。
- [ ] 检测到交互式挑战后立即终止自动升级和 stealth 重试。

验收标准：挑战页、登录页、正常商品页和内容过短页不会互相误判。

#### HC-006 修正单域名限流与请求合并

- [ ] 真正接入 `max_per_domain`。
- [ ] 限流键至少包含规范化域名；登录态请求建议使用 `domain + session_id`。
- [ ] 同一缓存键的并发请求使用 singleflight 合并。
- [ ] 域名规则可配置最小访问间隔、并发数和冷却时间。
- [ ] 限流等待必须支持取消和超时。

验收标准：同域名并发不超过配置，不同域名可以并发，相同 URL 只产生一次上游访问。

#### HC-007 增加挑战负缓存与熔断

- [ ] `CHALLENGE_NOT_SOLVED` 不再立即过期。
- [ ] 按 `domain + session_id` 记录 `next_allowed_at`。
- [ ] 冷却期间新请求直接复用已有活动挑战，或返回 `HUMAN_REQUIRED/COOLDOWN`。
- [ ] 429、403、挑战页使用不同冷却策略。
- [ ] 成功恢复后只清除对应会话的挑战熔断，不影响其他账号。

依赖：HC-002、HC-006。

验收标准：连续提交同一淘宝 URL 不会重复创建浏览器挑战或重复打上游。

#### HC-008 实现安全接管网关

- [ ] 提供一次性 HTTPS 接管 URL。
- [ ] 令牌包含/关联 `challenge_id`、`owner_id`、到期时间和随机 nonce。
- [ ] 令牌只保存摘要，成功使用、取消或过期后立即失效。
- [ ] 接入 noVNC/WebSocket 或等价的远程浏览器方案，显示并控制原浏览器页面。
- [ ] 限制地址栏、下载、剪贴板、文件上传、开发者工具和非目标域名导航。
- [ ] 关闭接管页面后，服务端能够判断断线并释放控制锁。

依赖：HC-004。

验收标准：未授权用户、过期令牌、重复使用令牌和跨挑战令牌均被拒绝。

#### HC-009 实现 IM 交互卡片与身份绑定

- [ ] 当抓取返回 `HUMAN_REQUIRED` 时发送人工验证卡片。
- [ ] 卡片包含站点、任务号、有效期，以及“打开验证”“取消任务”按钮。
- [ ] 接管令牌必须绑定 IM 平台用户 ID，不得只依赖链接保密。
- [ ] 验证成功、失败、取消和超时都向原会话推送最终消息。
- [ ] IM 重复投递必须幂等，不能重复创建挑战或重复恢复任务。

依赖：HC-004、HC-008。

验收标准：只有发起任务的用户或明确授权人员可以接管。

#### HC-010 实现异步暂停与恢复

- [ ] 原 MCP/HTTP 调用快速返回 `HUMAN_REQUIRED`，不得保持长连接等待用户。
- [ ] 保存恢复所需的 `job_id`、目标 URL、转换参数和页面句柄关联。
- [ ] 挑战通过后从原 `Page` 读取最终 URL 和 DOM，而不是重新请求目标 URL。
- [ ] 恢复过程使用状态条件更新，保证最多执行一次。
- [ ] 恢复成功后按现有 pipeline 转换、缓存和持久化。
- [ ] 恢复失败时生成明确错误码，并关闭 context。

依赖：HC-003、HC-004、HC-009。

验收标准：重复 webhook、重复 `poll` 或 IM 重试不会导致二次抓取和二次写入。

#### HC-011 强化登录成功验证

- [ ] 不再把“离开登录 URL”直接等同于登录成功。
- [ ] 先确认最终 host 属于适配器允许列表。
- [ ] 确认挑战标记不存在。
- [ ] 使用站点适配器确认登录态 DOM 或轻量授权页面。
- [ ] 只有验证成功后才封存 profile。

验收标准：跳到滑块页、错误页、第三方页均不会创建 ACTIVE profile。

#### HC-012 保持登录与抓取环境一致

- [ ] 每个账号绑定固定 profile 配置。
- [ ] 登录和抓取使用兼容的 Chromium 版本、启动参数、viewport、locale、timezone 和网络路由。
- [ ] 人工验证期间禁止 BrowserPool 的定期重启影响该 context。
- [ ] 同一 profile 串行使用，禁止两个进程同时打开。
- [ ] profile 过期或损坏时明确要求重新登录，禁止静默创建新身份。

验收标准：人工验证后的同一页面可继续抓取，下一次授权访问能复用已保存登录态。

### P1：上线前应完成

#### HC-013 安全与隐私加固

- [ ] 接管入口强制 HTTPS，并设置严格 CSP、`frame-ancestors` 和安全 Cookie。
- [ ] WebSocket 校验 Origin、用户身份、挑战状态和一次性令牌。
- [ ] 日志脱敏覆盖 token、Cookie、Authorization、二维码和截图内容。
- [ ] 浏览器 worker 与主服务网络隔离，只开放必要端口。
- [ ] 接管页面禁止访问内网地址和非 HTTP(S) scheme。
- [ ] 记录审计事件：创建、认领、验证、撤销、过期；不记录用户拖动轨迹。

#### HC-014 可观测性

- [ ] 指标：挑战触发率、人工完成率、平均接管时长、超时率、误判率。
- [ ] 指标按域名和挑战类型聚合，不使用账号、URL 查询参数作为高基数标签。
- [ ] 增加活动 challenge/context 数量和资源泄漏告警。
- [ ] 记录从抓取到挑战、从挑战到恢复的 trace 关联。

#### HC-015 运维与容量保护

- [ ] 配置最大活动挑战数和每用户活动挑战数。
- [ ] 达到容量时返回明确的 `CHALLENGE_CAPACITY_EXCEEDED`。
- [ ] 部署健康检查覆盖 Browser Worker、接管网关和消息队列。
- [ ] 进程滚动升级前停止接受新挑战，并给活动挑战提供完成窗口。
- [ ] 制定 Browser Worker 异常重启后的失败通知策略。

#### HC-016 文档与操作手册

- [ ] 更新 MCP 工具和错误码文档。
- [ ] 编写 IM 用户操作说明。
- [ ] 编写超时、接管失败、浏览器 worker 异常的排障手册。
- [ ] 说明单进程限制和多实例部署拓扑。
- [ ] 明确站点授权、服务协议和数据使用边界。

### P2：后续优化

#### HC-017 数据源路由

- [ ] 支持按域名配置 `official_api -> feed -> public_html -> authorized_browser` 的数据源优先级。
- [ ] 公开数据缓存与账号私有数据缓存隔离。
- [ ] 可使用已有陈旧缓存降级返回，并异步刷新。

#### HC-018 独立 Browser Worker

- [ ] 将浏览器生命周期从 MCP 服务拆到独立 worker。
- [ ] 使用 challenge 路由键确保控制流、验证流和恢复流落到同一 worker。
- [ ] worker 只接受签名后的内部命令。
- [ ] 支持受控排空，不承诺在 worker 崩溃后恢复内存中的页面。

## 6. 建议交付顺序

1. HC-001、HC-002：确定模型和迁移。
2. HC-003、HC-004、HC-005：建立页面移交及挑战生命周期。
3. HC-006、HC-007：先解决请求放大问题。
4. HC-008、HC-009：接入人工操作入口和 IM 卡片。
5. HC-010、HC-011、HC-012：打通验证、恢复和 profile 一致性。
6. HC-013 至 HC-016：安全、观测、容量和文档后上线。
7. HC-017、HC-018：按规模演进。

## 7. 上线验收门槛

- [ ] 所有 P0 任务完成。
- [ ] 测试文档中的 P0 自动化测试全部通过。
- [ ] 安全测试不存在高危或严重问题。
- [ ] 连续 100 次挑战生命周期测试没有残留 Page、Context、进程或有效令牌。
- [ ] 同一挑战在重复消息、重复 webhook 和并发恢复下最多恢复一次。
- [ ] 滑块触发后不再自动重试上游。
- [ ] 未授权用户无法查看或控制页面。
- [ ] 用户超时、取消、断线和服务关闭均能释放资源并收到明确结果。

## 8. 非目标

- 自动识别滑块缺口或生成拖动轨迹。
- 接入第三方打码服务。
- 代理池轮换、账号轮换或购买登录态。
- 绕过站点访问控制、服务协议或数据授权限制。
- 保证所有站点、所有挑战供应商都能通过人工接管恢复。
